
# Image Metadata API

## Описание

REST API сервис для хранения мета-информации об изображениях. Позволяет сохранять URL, размеры и теги изображений, а также выполнять поиск по тегам.

Данные хранятся в PostgreSQL. Связь между изображениями и тегами реализована через промежуточную таблицу (многие ко многим) — одно изображение может иметь несколько тегов, и один тег может быть присвоен нескольким изображениям.

## Стек

- Python 3.12, Flask, Flask-SQLAlchemy
- PostgreSQL 16
- Docker, Docker Compose

## Запуск

```
docker compose up --build
```

После запуска API доступен на `http://localhost:5000`.

## Эндпоинты

| Метод | Путь | Описание |
|-------|------|----------|
| `POST` | `/images` | Создать запись об изображении |
| `GET` | `/images` | Получить список всех изображений |
| `GET` | `/images?tag=landscape` | Получить изображения с указанным тегом |
| `GET` | `/images/{id}` | Получить информацию об изображении по ID |

## Примеры запросов

Создание изображения:

```
curl -X POST http://localhost:5000/images \
  -H "Content-Type: application/json" \
  -d '{"url":"https://example.com/photo.jpg","width":800,"height":600,"tags":["landscape"]}'
```

Все изображения:

```
curl http://localhost:5000/images
```

Фильтр по тегу:

```
curl http://localhost:5000/images?tag=portrait
```

Изображение по ID:

```
curl http://localhost:5000/images/1
```

## Принятые решения

**База данных.** Выбран PostgreSQL, поскольку задача предполагает хранение связанных сущностей (изображения и теги). Данные сохраняются в Docker volume `pgdata` и не теряются при перезапуске контейнеров.

**Связь изображений и тегов.** Реализована через промежуточную таблицу `image_tags`. При создании изображения теги ищутся в базе по имени — если тег уже существует, создаётся только связь, дубликат не появляется.

**Порядок запуска сервисов.** В `docker-compose.yml` для контейнера PostgreSQL настроен healthcheck (`pg_isready`). Приложение стартует только после того, как база готова принимать соединения.

**Валидация.** При отсутствии обязательных полей (`url`, `width`, `height`) или при невалидном JSON API возвращает `400`. При запросе несуществующего ID — `404`.
